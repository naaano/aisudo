# aisudo

Secret broker and risk-aware gate for AI agents and automation tools. Protects sensitive credentials and catches genuinely dangerous actions. Lets common operations flow without interruption.

## Philosophy

aisudo is **not** a gate for every command. Agents already run with broad access and that's fine. aisudo exists to:

- Protect tokens, credentials, and secrets from uncommon or risky access
- Catch genuinely dangerous actions (deploy, publish, destructive ops)
- Let common, repetitive calls flow without approval
- Stay agnostic to specific agents, editors, IDEs, and tools
- Integrate with existing platform secret stores (macOS Keychain, etc.)

_Avoid_: "approve every call", "sudo for bash", TCC-style permission spam

## Language

**App**:
The process that directly calls aisudo via CLI or local socket. Identity is self-reported (`name`, `pid`, `cwd`) and verifiable later via platform signing. Not the originating IDE or shell — just the immediate caller.
_Avoid_: agent, tool, IDE (these are roles, not identities)

**Action**:
A classified operation an App attempts. Defined by a finite taxonomy: `exec`, `secret`, `file-read`, `file-write`, `network`. Apps provide the action kind plus structured metadata. Policy rules match on kind and metadata.
_Avoid_: request type, operation, command

**Policy**:
The single source of truth for what is allowed, denied, or requires human approval. Ships with sensible defaults. Users tune it by describing their workflow to an agent, which proposes changes. Humans always approve/reject policy changes.
_Avoid_: ruleset, config, heuristics engine

**Unknown Action**:
An action that matches no policy rule. Defaults to `allow` and logs to audit. Exception: `secret` access defaults to `ask` when unmatched, because secrets are the primary protected resource. Unknown actions can be captured into policy rules via proposals after the fact.
_Avoid_: unclassified, default-deny

**Policy Proposal**:
A set of proposed policy changes generated by an agent based on user intent. Presented as a simple diff for human approval. Agents write proposals; humans decide.
_Avoid_: rule suggestion, policy draft

**Secret**:
A credential, token, API key, JWT, or session token that aisudo protects. aisudo owns the metadata (name, kind, scope, risk, expiry) but not the value. The value lives in a pluggable backend.
_Avoid_: credential, password (too narrow)

**Secret Registry**:
The metadata layer that knows what secrets exist, what they're for, and how sensitive they are. Not a vault — a classification and gating layer. Entries point to a backend for the actual value.
_Avoid_: vault, secret store

**Secret Backend**:
Pluggable storage for secret values. Implementations: Keychain, DPAPI, 1Password CLI, or aisudo's own encrypted file. Backend is abstracted — aisudo doesn't care where the value lives.
_Avoid_: provider, store

**Severity**:
How urgently the human needs to respond. Three tiers:
- **Info**: No notification. Logged only. Common operations that touch nothing sensitive.
- **Alert**: Silent notification, review later. Actions with elevated risk but no secret/sensitive access.
- **Critical**: Loud notification, wake screen. Secret access, sensitive file access, destructive operations.
_Avoid_: risk level, priority (overloaded terms)

**Default Policy**:
The sensible out-of-the-box policy that ships with aisudo. No configuration session required — it works from first run. Users evolve it over time via agent proposals. This is the "built-in security layer" expressed as policy, not a separate heuristic engine.
_Avoid_: built-in heuristics (it's just the default policy)

**Decision**:
The outcome of a policy evaluation. Four values:
- `allow` — run silently, log
- `alert` — run, silent Telegram notification, log
- `ask` — block, loud notification, wait for human approval
- `deny` — block silently, log
_Avoid_: verdict, outcome

**Transport**:
The delivery channel for notifications and approval responses. Primary: Telegram Bot API (one-tap inline buttons, zero install). Fallback: ntfy, local socket. Transport is pluggable — aisudo doesn't depend on any single service.
_Avoid_: channel, relay

**Grant**:
A temporary approval for a specific action+app+context combination. Two types: `once` (approve this now, no persistence) and `scoped` (approve this for a duration). "Always allow" is a policy rule, not a grant — the agent proposes it.
_Avoid_: permission, allowance, temporary rule

**Integration**:
How apps connect to aisudo. Two paths: explicit (`aisudo exec`, `aisudo run`) for reliable agent-native integration, and shell hooks (`aisudo init`) for transparent terminal interception. Shell hooks are the easy on-ramp but fragile for non-interactive agents. Explicit is always reliable.
_Avoid_: wrapper, shim

**Agent-Centric UX**:
The agent is the primary UI for aisudo. Users prompt their agent ("install aisudo", "make a policy for me") and the agent drives setup, policy creation, and management. aisudo provides CLI primitives; the agent orchestrates them. The CLI is the backend API, not the user interface.
_Avoid_: CLI-first UX, manual config

**Secret Scan**:
An onboarding step during `aisudo init` (or `aisudo scan`). Scans the filesystem for exposed secrets — .env files, hardcoded tokens, API keys in source code — and offers to migrate them to the secret backend. The agent drives the scan and proposes what to move. User approves each migration.
_Avoid_: audit, cleanup

**Audit Log**:
Append-only record of every request aisudo processes. JSON lines at `~/.aisudo/audit.log`. Each entry has: id, time, app, action, summary, decision, matched rule, severity. Agents read it to propose policy improvements. Humans tail it to see what's happening.
_Avoid_: event log, history

**Daemon**:
Long-running background process that listens for requests, evaluates policy, sends Telegram notifications, and waits for approvals. Runs as a platform service (launchd on macOS, systemd on Linux, Windows Service) so it auto-starts and restarts on crash. Daemon down = all workflows blocked, so reliability is critical.
_Avoid_: server, service (overloaded)

## CLI (v0.1)

- `aisudo init` — first-run setup
- `aisudo daemon` — run the daemon
- `aisudo exec -- <cmd>` — gate a command
- `aisudo run -- <cmd>` — inject secrets and run
- `aisudo ask secret <name>` — request secret access
- `aisudo scan` — find exposed secrets
- `aisudo policy` — show current policy
- `aisudo policy propose` — apply agent-generated policy
- `aisudo audit tail` — follow audit log
- `aisudo audit mine` — my app's activity
